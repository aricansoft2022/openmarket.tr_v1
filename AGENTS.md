# Agent operating contract

## Authority order

1. `spec.md` and the SHA-verified `spec.full.md` generated by `npm run spec:materialize`
2. Explicit user instructions
3. `DECISIONS.md`
4. `ARCHITECTURE.md`, `DATA_MODEL.md`, `SEED_CATALOGUE.md`, `TEST_PLAN.md`
5. `STATUS.md` and `HANDOFF.md`
6. `DESIGN_REFERENCE.md`

Do not infer new business rules from the prototype.

## Session start

1. Run `npm run spec:materialize`, then read `spec.full.md`, `STATUS.md`, `HANDOFF.md` and recent `DECISIONS.md` entries.
2. Inspect the branch, open PR, migrations and working tree.
3. Run the verification commands recorded in `HANDOFF.md`.
4. Continue only the exact next tasks or update the handoff before changing scope.

## Session end

1. Update `STATUS.md` with phase, completed work, tests and blockers.
2. Update `HANDOFF.md` with exact next tasks and commands.
3. Append accepted and rejected decisions to `DECISIONS.md`.
4. Record migration and seed changes.
5. Leave no undocumented partial migration, secret, manual dashboard step or failing test.

## Engineering constraints

- Keep a modular monolith until measured load or security isolation requires a split.
- Runtime PostgreSQL access goes through Hyperdrive; migration tooling uses direct Neon credentials.
- Never store all dynamic product attributes in an unvalidated JSON blob.
- Never expose private R2 objects with permanent public URLs.
- Every privileged mutation writes an audit event in the same transaction where possible.
- Queue publication must use an outbox pattern for state-changing workflows.
- No paid ranking, hidden supplier score, donor advantage or unverified trust badge.
